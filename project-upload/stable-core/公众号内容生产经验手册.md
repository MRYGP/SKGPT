## 一、初次写作的教训（001-003第一轮）

### 教训1：单案例撑不住一篇文章

每篇至少一组正反对比。一个做对的，一个做错的。读者记不住道理，但记得住对比。

### 教训2：灵感来源不能当配角

如果你的灵感来自一个有力的原型（AlphaFold、拼多多、巴菲特），不要把它缩成一句引用。展开它，让它成为文章的骨架。读者通过故事记住工具。

### 教训3：工具类文章必须"跑给你看"

每个工具/公式类文章，必须有一个"跑给你看"的实战段落。不能只讲原理，要当场演示。

### 教训4：用判断力开场，不用自我介绍开场

读者因为"你说了一句我没想到的话"而关注你，不是因为"你是谁"。

### 教训5：每篇必须跟上一篇"接上"

每篇至少在两个地方引用前面的文章。读者可以从任何一篇进入，但进入后能感觉到"这背后有一个体系"。

### 教训6：预告下一篇要具体，不能泛

"下一篇更精彩"不是预告，是废话。要么是让人好奇的事实，要么是明确的承诺。

### 教训7：系列文章之间要有"角色分工"

每篇有明确的"读者读完后带走什么"。如果两篇的角色重叠，合并成一篇。

### 教训8：不要把理论底稿直接当文章发

理论底稿是弹药库，文章是步枪。每次只装一颗子弹。底稿里没用上的素材不是浪费，是存着以后用的。

### 教训9：不能连续讲道理

连续两段解释/分析后，必须插入故事、数字或对比。读者可以忍一段道理，忍不了三段。

## 二、改稿和发布的教训（001六轮迭代，v2→v6）

### 教训10：产品案例必须联网验证，不能凭记忆

**血泪案例**：001-v2里写"票圈：做长图拼接，2亿用户"——三处全错。

**铁律**：引用任何产品作为案例时——名称、功能、数据、现状，每一项都要查。拿不准的宁可不写。

### 教训11：金句会被多轮改写"磨掉"

**铁律**：底稿里你自己写的尖锐句子是圣物。发给写作GPTs之前，先标出来告诉它"这几句不能改"。改完后第一时间检查还在不在。

### 教训12：数据不能全删，留2-3个锚点

**铁律**：底稿里10个数字，成稿留2-3个最有冲击力的。用聊天口吻带出来。

### 教训13：结尾三元素一个都不能少

评论区问题 + 下篇预告 + 系列标识。缺一不可。

### 教训14：3000字是上限，不是目标

成稿超过3500字就回去砍。

### 教训15：读者GPTs的标准要校准——这是公众号不是论文

A类硬伤零容忍，B/C类按公众号标准放宽。传播力测试权重不低于事实核查。

## 三、内容策略教训（001发布后+002规划的反思）

### 教训16：产品是触发器，现象是入口——两者相互，但文章从现象开头

**核心认知**：产品和现象是相互的。更多的时候，是一个产品让我们好奇一个现象/问题。但文章的入口永远是那个让普通人也好奇的现象，不是产品名。

**选题工作流**：
```
雷达发现一个有趣的产品（触发器）
    ↓
问自己：这个产品让我好奇什么现象/问题？
    ↓
那个现象/问题，就是文章的标题和开头
    ↓
产品是回答那个问题的证据，出现在文章中间
```

**铁律**：写标题和开头时问自己："一个不认识这个产品的人，会点进来吗？"如果不会，说明你还没找到那个让普通人也好奇的现象入口。

### 教训17：雷达不只是产品库，是选题库

**核心认知**：真正值得深度拆解的AI产品，一周甚至一个月才出来一个。30篇不可能全是产品拆解。

雷达扫描的过程本身会产出大量素材：
- 趋势观察（"这个月突然多了5个做XX的产品"→ 方向文章）
- 某类产品集体死了（"这三个产品同一个月关了"→ 死法文章）
- 信号源本身的评价（→ 方法论文章）

**铁律**：每次跑完雷达扫描，问自己：
1. "哪个产品让我好奇了？好奇什么？"（→ 选题触发器）
2. "这次扫描本身有没有产出一个'现象'？"（→ 直接选题）

## 四、多版本迭代教训（004-007实战总结）

### 教训18：多版本迭代时，必须主动追溯之前版本的遗漏

**铁律**：在任何版本定稿前，逐一检查前面所有版本，明确问：
- "这次删掉的，有没有功能上独立有价值、只是位置不好的内容？"
- 如果有，不是删，是找更好的位置安置它。

### 教训19：被采访对象的原话比喻，与作者金句同等保护

**扩展圣物定义**：
- 原圣物：作者自己写的尖锐判断句
- 新圣物：被采访对象/创始人说的有画面感的比喻或判断

**铁律**：底稿整理时，两类圣物都标【圣物】：
- 【圣物-作者】：自己写的尖锐原话
- 【圣物-原话】：创始人/被访者的核心比喻

### 教训20：删段落前，先检查段落内有没有功能不同的内容混在一起

**铁律**：删一段之前，问两个问题：
1. 这段的功能是什么？
2. 这段里有没有功能不同的内容被混在一起？如果有，分开处理，不要一刀切。

### 教训21：不拆篇的判断标准

当文章感觉"太重"时，第一反应不是拆成两篇，而是问：

**文章里的第二个强洞察，是第一个洞察的证据，还是独立的另一个洞察？**

- 是证据 → 不拆，砍掉冗余内容，保留核心
- 是独立洞察 → 考虑拆篇，两篇各自完整

### 教训22：多AI同时给反馈时，共识不等于编辑正确

**铁律**：面对多AI反馈时，对每条建议独立问：
1. 这条建议采纳后，"那一刀"变锋利了还是变钝了？
2. 这条建议解决的是读者的问题，还是让分析显得更完整？
3. 写作宗旨（让陌生人停下来记住一句话）是否支持这个改动？

如果三条里有一条是否定的，不采纳。共识只是参考，编辑判断是最终权威。

### 教训23：证据等级标注只属于底稿，不属于发布版

**来源**：007多版本迭代实战（2026-04-02）

**核心认知**：
- `（A级·xxx）`、`（B级·xxx）`、`（C级·推断）` 是写作过程中的质量控制工具
- 这些符号的读者是作者自己和协作AI，不是公众号读者
- 发布版里出现这些标注，会让文章显得像"内部文件"，破坏阅读感

**铁律**：底稿保留所有证据等级标注（用于质控）。发布版/投稿版一律去掉，按以下规则处理：

| 底稿写法 | 发布版写法 |
|---------|----------|
| `40%的企业……（B级·BCG 2025年报告）` | `BCG 2025年调研：40%的企业……` |
| `估值21亿美金（A级·BusinessWire）` | `估值21亿美金（BusinessWire，2025年11月）` |
| `窗口判断：12-18个月（C级·推断）` | `窗口判断：12-18个月。（这是推断，不是数据）` 或直接去掉括号 |
| `……（X级·未验证）` | 整段删掉或不写 |

**标注什么时候消失**：
- 走写作工坊流程时：告诉写作GPTs"去掉所有A/B/C/X级标注，来源用自然语言保留"
- 整理投稿版时：直接全文替换，见 `cases/2026/投稿版/` 目录

---

### 教训24：找不到那一刀，就不开始写底稿

**来源**：写作前准备SOP推演（2026-04-03）

**核心认知**：
底稿写了一半发现"没有一个让人记住的点"，不是写作能力的问题。是因为尚未确定那一刀就开始写了。

那一刀写不出来，可能的原因只有两个：
1. 材料不够——对产品的理解还停在表面
2. 理解够了但没有公式化——用Phase 0C的过程强制输出就能找到

**铁律**：
- 底稿是那一刀的展开，不是找那一刀的地方
- 写不出那一刀 → 进入Phase 0A/0B补课，不是强行写下去
- 那一刀确认了，底稿通常写得很快——因为你知道这篇文章只有一个目的

## 六、定稿前追溯清单

> 每次完成最终版本前，额外过这3个问题：

1. **前版本追溯**：之前的版本里，有没有被删掉但实际独立有价值的内容？逐段对照。（教训18）
2. **段落功能检查**：本次删掉的段落里，有没有功能不同的内容被混在一起删掉了？（教训20）
3. **圣物完整性**：两类圣物（作者原话+被访者原话）全部在位？（教训11+19）

## 七、标准流程（从002开始）

```
Phase 0A：材料收集（15-30分钟）
   搜产品基本面——规模/收入/团队/融资
   搜创始人原话——访谈/演讲/博客
   搜独立第三方数据——Sacra/TechCrunch/行业报告
   搜失败对比产品——同类死亡案例
   搜中国市场现状——对应的中国产品/空白
   数据不足 → 触发深度研究员

Phase 0B：10维度深度拆解（1-2小时，A型文章必做）
   填写 product-teardown-template-v4.0.md 全部10个维度
   存为深度底稿：cases/2026/深度底稿/NNN-公司名-拆解底稿.md
   所有断言标注证据等级（A/B/C/X）
   ❗ B/C型文章不需要完整拆解，过相关维度即可

Phase 0C：那一刀确认（15分钟）
   必须写出一句话：
   「如果这篇文章只说一件事，让没见过这个产品的人10秒内决定'值得看'，那件事是：___」
   写不出来 → 回到Phase 0A补材料，不进入Phase 0D
   写出来了 → 这就是发给写作GPTs的「那一刀」

Phase 0D：理论框架决策（15分钟）
   选一个能放大那一刀的框架：
   • 产品有飞轮/规则制定 → RRF（先搜「RRF案例验证库」）
   • 产品核心是用户关系 → 财富公式
   • 需论证需求高频 → 推演公式
   • 产品切被忽视的用户 → 破坏式创新
   • 产品逻辑在当时看起来是错的 → 从0到1
   • RRF不适用于：纯硬科技、专业服务、消费硬件、强监管行业
   • 做了RRF分析 ≠ 文章里要讲RRF（框架是分析工具，不是输出内容）

          ↓

1. 思考Project出底稿
   ↓ 标出【圣物-作者】（不能改的作者金句）
   ↓ 标出【圣物-原话】（被访者/创始人的核心比喻）
   ↓ 标出【锚点】（保留的关键数字）
   ↓ 保留（A/B/C/X级）标注供质控用
2. 发给写作GPTs改写
   ↓ 附上文章类型和大纲位置
   ↓ 告诉它"那一刀"是什么
   ↓ 告诉它"去掉所有A/B/C/X级标注，来源用自然语言保留"
3. 拿回改写稿，立即检查：
   ✓ 两类圣物还在不在？
   ✓ 锚点数字还在不在？
   ✓ 结尾三元素在不在？
   ✓ 字数在3000以内？
   ✓ A/B/C/X标注已全部消失？
4. 发给读者GPTs检测
   ↓ 看事实核查报告
   ↓ 看传播力测试（截图句几个？转发配文是什么？）
   ↓ 看作者vs读者视角比例
5. 根据反馈修改
   ↓ 对每条建议问：采纳后"那一刀"更锋利了还是更钝了？
   ↓ 多AI共识不等于正确，从写作宗旨出发独立判断（教训22）
6. 定稿前过追溯清单（第六节，3个问题）
7. 过发布前检查清单（基础22条 + **全局第26条**；诊断空白/招募型另加第23–25条；机会判断类另加第27条）
8. 发布
```

## 九、身份文与规律文写法教训（006-010实战总结）

### 教训25：身份文也不能从"我"开始

**来源**：009两版对比（旧版「50岁。小老板。」vs 新版「两类人最不容易被AI替代」）

**铁律**：即使是介绍自己的文章，开头也必须是让陌生人想继续读的观察或现象。"我是谁"永远不是入口，观察才是。

判断标准：把开头第一句话给一个完全不认识你的人看，他会因为"这个判断有意思"继续读，还是因为"这个人在介绍自己"跳过？

### 教训26：公开预告不能反映内部决策进度

**来源**：010原结尾"下一篇：我们决定押注老年赛道"被推翻

**错误逻辑**：把内部推演结论（红队PK还没跑）当成公开承诺写进文章。

**铁律**：文章预告只写"接下来会做什么"，不写"我们内部已经决定了什么"。还没确定的事绝对不要出现在公开文章里——读者会记住，会追问，会让你被自己的话绑住。

悬念（"找到了你们会第一个知道"）比承诺（"下一篇押注XX赛道"）更好用，也更安全。

### 教训27：规律类文章用"奇怪的事"不用"三条规律"

**来源**：010选择B型（复盘叙事）而非A型（规律提炼）的实战验证

**核心认知**：
- "我发现了三条规律" → 课堂感，读者是学生
- "我发现了一件奇怪的事" → 旅程感，读者是同行者

规律是结论，奇怪的事是过程。读者愿意跟着走过程，不愿意被喂结论。

**铁律**：凡是提炼规律类的文章，用亲历者视角（"我发现了一件奇怪的事"）而不是老师视角（"总结三条规律"）。即使规律本身完全一样，包装方式决定读者是否愿意跟进来。

### 教训28（工具层）：bash_tool写不到用户的Windows文件系统

**来源**：009/010文件反复写入失败的实战教训

**核心认知**：bash_tool跑在Claude的Linux环境里，Python的open()写`D:\sk\...`路径会报成功但实际没有写入用户电脑。

**铁律**：所有需要写入用户D盘的操作，必须用`Filesystem:write_file`工具，不能用bash_tool里的Python文件操作。判断方法：路径是`D:\`开头的，用Filesystem工具；路径是`/home/claude/`或`/tmp/`开头的，用bash_tool。

## 十、产品复制与创业方法论（2026-04-15）

### 教训29：复制不是终点，模式叠加才是壁垒

**来源**：产品复制方法论讨论（2026-04-15）

**核心认知**：

- **Layer 1 直接复制**：看到海外产品→做中文版，3个月后10个竞品出现，没有优势。
- **Layer 2 横向复制**：提取产品的核心模式，应用到相邻赛道，暂时领先但模式同样可被复制。
- **Layer 3 模式叠加（目标层）**：把多个成功产品的换题模式叠加到一个新场景。组合罕见，壁垒来自叠加的稀有性。

**示例**：MYHAIR AI诊断模式 + Yoodli「帮用户练」模式 = 皮肤健康训练AI，这个组合大概率没人做过。

**铁律**：评估一个新方向时，除了问"这个产品值不值得复制"，还要问"我能叠加哪几个换题模式，让它变成别人难以复制的组合"。

### 教训30：Gamma/Lovable/Yoodli/Paid不是"要不要复制的产品"，是可复用的换题模式库

**来源**：产品复制方法论讨论（2026-04-15）

**核心认知**：这4个产品放在"C/B级难复制"分类里是不完整的。它们的真正价值是提炼出了4个可叠加到任何产品上的底层模式（P1-P4）：

- **P1 产出物即分发**（来自Gamma）：用户产出物天然携带品牌，叠加到任何用户会分享产出物的工具上。
- **P2 门外汉上场**（来自Lovable）：服务被现有工具排除在外的人，叠加到任何专家垄断的领域。
- **P3 帮用户练**（来自Yoodli）：从替用户做到帮用户练，叠加到任何「知道≠会做」的场景。
- **P4 激励对齐**（来自Paid）：按结果收费而非按时间/功能，叠加到任何价值难以度量的专业服务。

**铁律**：每次深度推演新产品时，先问能否叠加P1-P4中的某几个模式。叠加点越多越独特，护城河越深。详细定义见 `core/产品评估决策清单-v1.0.md` 第四节。

## 十一、框架应用陷阱（2026-04-18，Cactus拆解实战）

### 教训31：框架反向驱动——先有框架再找例子，是分析最隐蔽的错误

**来源**：Cactus拆解中国机会分析（2026-04-18）

**犯错过程**：
提炼出P5「新客户零等待原则」之后，开始在中国市场里「找满足三条件的场景」（紧急+新客户+零替代成本），找到了老人陪诊/宠物急诊/婚庆应急，认为「P5成立，可以切入」。

被用户直接点出：这是硬套。

**错误的本质**：
P5框架有一个隐含的**基础设施前提**，在提炼框架时没有显式写出来：

> 消费者能通过搜索独立发现个体服务商，然后直接拿到这个人的电话打过去。

美国：Google本地搜索→个体师傅网站/Google Business→电话号码→打过去。这条路在美国是主干道。

中国：高德/大众点评/美团已截断发现层，消费者在联系服务商之前就已经经过了平台过滤。「紧急的新客户直接打个体师傅电话」这个前提在中国城市市场几乎不存在。

找到的那些「紧急场景」（宠物急诊/老人陪诊）确实有紧急性，但消费者找到服务商的路径仍然是平台——P5的根本前提不满足，三条件满足与否都无意义。

**铁律**：
任何框架都有隐含的**基础设施前提**——那些框架成立所依赖的、往往被认为理所当然的环境条件。在把框架应用到新市场之前，必须先问：

> 「这个框架成立所需要的基础设施/环境前提，在目标市场里存在吗？」

不是先问「这个市场有没有满足三条件的场景」，而是先问「这个框架的底层土壤在这里存在吗」。土壤不存在，再好的种子也种不活。

**判断方法（通用）**：
提炼出一个框架/模式后，显式列出：
1. 这个框架在原始案例里成立，依赖了哪些「理所当然」的基础设施？
2. 目标市场里，这些基础设施存在吗？
3. **如果不存在，直接结论「不适用」，不要试图找边缘案例勉强凑合。**

**框架反向驱动的信号**：你在积极地「寻找例子」而不是在「验证前提」。当你开始找例子的时候，说明你已经在为结论找理由，不是在推导结论。

**更广泛的应用**：
这不只是「中国市场分析」的问题。每次把任何框架（破坏式创新/P1-P5/RRF/诊断空白四条件）迁移到新场景时，都要先做这个「前提检验」，而不是直接跳进「找满足条件的例子」。

## 十二、真实产品外部体检教训（2026-04-29）

### 教训32：真实产品拆解不是点评，而是外部体检

**来源**：Coomia 触发的“真实产品外部体检”场景（2026-04-29）

**核心认知**：
当朋友/创业者发来早期产品时，最容易犯的错是把拆解写成“评审点评”：要么像投资人打分，要么像评论区吐槽。两者都不解决对方真正问题。

真实产品拆解的目标应是三件事：
1. 先确认已成立价值（避免误伤）
2. 找到外部误解和叙事错位（为什么用户看不懂）
3. 推演可迁移方向与输出策略（私下建议 / 脱敏公开 / 公开点名）

**铁律**：做真实产品拆解时，先问“这份输出能让对方下一步更清楚吗”，而不是“这段点评够不够锋利”。

## 十三、诊断空白与招募信号（2026-04-29 · 013发布后复盘）

### 教训33：文章不能只把问题写清楚，还要把可做的位置写清楚

**来源**：013 口腔发布后复盘——「信任空白」写清了，但「治疗方案第三方审查层」等商业突破口前置不足，文章偏弱招募、偏行业分析。

**核心认知**：诊断空白、MTP构思招募、真实产品外部体检、寻找同行者/资源人的文章，如果只停留在「行业结构性有问题」，读者读完不知道「作者主张的卡位/产品形态是什么」，就不会产生「我该不该联系这个人」的动力。

**铁律**：前半篇就要让读者能复述——空位在哪、与旧解法差在哪、钱可能从哪来、谁在找谁。

**过稿自检（新增一问）**：
- **这篇文章有没有在前三分之一打出「真正空着的位置」（可做 product slot），而不只是把痛点和行业失灵写透？**

---

### 教训34：招募型文章必须像机会说明书，不只是问题分析

**来源**：013 口腔文章发布后复盘正式入库（2026-04-30）。

013 已证明「口腔是信任空白」，但「治疗方案第三方审查层」这一**可做位置**没有足够早、足够重地打出时，文章读起来更像行业问题分析，**招募信号弱**。

这类文章的目标，不只是让读者觉得「这个行业有问题」，而是让潜在同行者、行业内部人、产品人、资源人看到：**这里可能有一个值得验证的新位置**。

**所有文章**必须在标题后加入 **「先说结论」**（见 **教训35**）。诊断空白、MTP、真实产品外部体检、机会招募类还须在该模块与正文中落实 **真位置**（见教训33–34与类型G）。

**「先说结论」必须包含五件事**：

1. **真问题**（表面在吵什么，真正卡在哪）；  
2. **真位置**（市场空着的可做 slot / 新角色 / 新交付层）；  
3. **可能商业模式**（付费时刻或收费动作的一句钩子）；  
4. **现有玩家为什么没占住**（利益结构 / 模型 / 监管 / 能力择要）；  
5. **我们要找谁**（可对号入座的人群 + 要验证的最关键假设）。

**判断标准**：懂行人读完**前三分之一**，应当能说出「这里可能有一个什么新产品 / 新公司 / 新角色」。若只能说「这个行业很乱 / 这个问题很大」，却说不出「这里空着什么位置」，**文章未过关**。

**模板位置**：`content/article_template.md` **全局规则「先说结论」** + **类型G · 机会招募型文章**（招募叠加五件事与前三分之一）。

## 十四、全局写作门槛（2026-04-30）

### 教训35：所有文章必须开头给「先说结论」

**来源**：013 口腔文章**发布后修订版**：补上「先说结论」后，读者能在开头判断——这篇发现什么问题、什么机会、想找什么人；对比发布版可看出突破口后置的损失。

三湘问道当前文章**不只是表达观点**，也在释放判断力、招募同行者、筛选资源人；陌生人只会给标题和开头几十秒。

**铁律**：以后**每一篇**正文都必须在 **`#` 标题之后**、**第一节之前**加入 **`## 先说结论`**。它不是论文摘要，而是 **判断卡 / 机会卡**。

**模块必须让读者在约 30 秒内知道**：

1. 这篇**不是在**说什么；  
2. **真正问题 / 核心判断**是什么；  
3. **真正空着的位置 / 核心机会 / 可迁移结构**是什么（拆解类则写清「这个案例对我们有什么用」）；  
4. **为什么它可能成立**；  
5. **想找谁 / 读者为什么要继续看 / 下一步要验证什么**。

**判断标准**

- **外行**读完：应当知道「这个问题为什么是真的」。  
- **懂行人**读完：应当知道「这里为什么值得讨论」。  
- **潜在同行者**读完：应当知道自己是否该联系作者。

**终稿门槛**：没有「先说结论」→ **不得进入终稿**。详见 `content/article_template.md` **v2.5**（全局「先说结论」+ 机会型「时间变量推敲」）；写作工坊 / 第一读者已同步强制。

## 十五、机会与时序（2026-04-30）

### 教训36：机会判断必须加入时间变量，但不能写成未来焦虑

**来源**：014 糖尿病前期文章复盘——若干细胞疗法、GLP-1、CGM、家用检测、AI 推理成本与端侧普及等 **未来 3–5 年变量**，可能 **改写**（而非简单消灭）同一赛道上的可做位置。

SK 若只看当前痛点与当前竞品空位，容易把机会写成**静态命题**，一旦被技术或监管迁移打脸，整篇信誉受损。

**正确用法**是看变量会如何：

- **打开**机会；
- **压缩**窗口；
- **改写**产品形态；
- 或在极端情况下 **杀死**当前叙事。

**铁律**

- **拆解**：`core/product-teardown-template.md` **§9.5 时间变量推敲**（四路决策前必填）。
- **真实产品外部体检**：`core/SKILL-真实产品外部体检与机会推演SOP.md` **Step 6.5**。
- **机会型文章**（类型G/H、诊断空白招募）：`content/article_template.md` **类型G第8步**，**150–300 字**，**只写 1–2 个最关键变量**。
- **第一读者**：`meta/第一读者-系统指令.md` **时间变量检查**。

**写作纪律**：不把文章写成未来预测报告；时间变量只用于 **校准判断**，不用于冲散主线；**不因未来可能变化就否定当前真实痛点**，也不因痛点强就忽略明显 **杀死变量**。

- 读者能否说出这篇在讲什么；
- 能否说出作者判断；
