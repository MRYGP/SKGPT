# SKILL：真实产品外部体检与机会推演 SOP

> 建议入库路径：`core/SKILL-真实产品外部体检与机会推演SOP.md`
>  版本：v0.2 · 2026-04-30（新增 Step 6.5 时间变量推敲）
>  适用场景：拆一个真实早期产品，既看到优点，也看到问题，并从中推演可迁移行业机会，最后转成公众号文章 / 私信建议 / 深度研究附件 / 产品招募入口。

------

## 一句话定义

**这套 SOP 不是“点评别人产品”，而是用外部视角做三件事：看见它做对了什么，指出它可能看不到的盲区，再从它身上推演出可迁移的新方向。**

------

## 0. 适用场景

适合这些情况：

1. 朋友、微信好友、读者、创业者发来一个产品，希望你看看；
2. 你发现一个小团队 AI 产品，觉得有启发；
3. 想写公众号文章，但不想只写“远处大案例”；
4. 想用真实产品展示 SK 的价值；
5. 想从一个产品里推演“我们能不能做类似结构的新方向”。

不适合这些情况：

1. 产品资料极少，只能靠猜；
2. 对方明确不希望公开；
3. 涉及未公开商业机密；
4. 你只想写一篇批评文；
5. 你没有时间做事实核验。

------

## 1. 总流程

```
Step 0：确认边界
      ↓
Step 1：收集公开资料
      ↓
Step 2：提炼第一层价值
      ↓
Step 3：确认它做对了什么
      ↓
Step 4：找外部误解 / 叙事错位
      ↓
Step 5：判断商业模式与交付风险
      ↓
Step 6：抽象可迁移结构
      ↓
Step 6.5：时间变量推敲
      ↓
Step 7：做小型深度研究
      ↓
Step 8：收束到一个具体方向
      ↓
Step 9：生成四类输出
      ↓
Step 10：决定是否公开、脱敏或点名
```

# Step 0：确认边界

## 目标

先判断这次拆解是：

- 私下建议；
- 公开文章；
- 脱敏案例；
- 点名共创；
- 内部研究；
- 产品机会推演。

## 操作

先问自己三个问题：

1. 这个产品和我是什么关系？
2. 是否需要对方同意才能公开？
3. 这次输出的目的是什么？

## 判断规则

| 情况                       | 处理方式                   |
| -------------------------- | -------------------------- |
| 对方是朋友 / 微信好友      | 默认先私发，不直接公开点名 |
| 产品完全公开，但对方是熟人 | 建议先征求同意             |
| 对方同意点名               | 写“公开体检版”             |
| 对方不同意或没回复         | 写“完全脱敏版”             |
| 涉及未公开资料             | 不写进文章，只做私下建议   |

## 标准话术

```
我最近在搭一个产品分析框架，想用外部视角帮早期产品做一次体检。

我会先看你们做对了什么，再看外部用户可能在哪里误解你们，最后也会从这个产品里推演一些可迁移的机会。

如果后面写公开文章，我会先让你们纠正事实错误。你们不想点名，我就完全脱敏。
```

# Step 1：收集公开资料

## 目标

不凭印象评价，先建立最小证据包。

## 必收资料

| 类型       | 具体内容                                   |
| ---------- | ------------------------------------------ |
| 官网       | 首页、产品页、定价页、案例页               |
| 文档       | Docs、API、部署说明、安装流程              |
| GitHub     | README、star、fork、release、issue、commit |
| 创始人内容 | 博客、推文、访谈、视频、朋友圈截图         |
| 用户证据   | 客户案例、评论、使用截图、demo             |
| 竞品资料   | 相邻产品、替代方案、国外同类               |

## 输出格式

```
## 公开资料包

产品名：
官网：
文档：
GitHub：
创始人文章：
客户案例：
定价：
主要宣传语：
主要功能：
证据缺口：
```

## 证据分级

| 等级 | 说明                                   | 用法           |
| ---- | -------------------------------------- | -------------- |
| A    | 官网、文档、公开价格、GitHub、真实截图 | 可直接引用     |
| B    | 权威媒体、行业报告、第三方评测         | 可辅助判断     |
| C    | 合理推断、类比、间接估算               | 必须标注“推测” |
| X    | 无法验证、听说、创始人口头说法         | 不进公开判断   |

# Step 2：提炼第一层价值

## 目标

把产品从“团队自己的大叙事”压回“用户第一眼能听懂的一句话”。

## 必答问题

1. 它到底帮谁？
2. 在什么场景下？
3. 解决什么具体问题？
4. 用户为什么现在不用旧方法？
5. 第一层价值能不能 10 秒说清？

## 输出模板

```
## 第一层价值

不要这样说：
“这是一个 AI 驱动的企业级智能平台。”

应该这样说：
“它帮 XX 人，在 XX 场景下，用 XX 方法，解决 XX 具体问题。”

本产品第一层价值：
它帮【具体用户】，在【具体场景】下，解决【具体卡点】。
```

## 检验句

如果这句话里出现：

- 一站式；
- 全流程；
- 平台；
- 生态；
- 赋能；
- 重塑；
- 智能中枢；

就要重新写。

# Step 3：确认它做对了什么

## 目标

先找优点，不要一上来挑刺。

这一步很重要。
 如果你看不到产品做对了什么，后面的批评没有信用。

## 四类优点

| 类型   | 问题                                           |
| ------ | ---------------------------------------------- |
| 真痛点 | 它是不是抓住了一个真实、具体、高频的问题？     |
| 真约束 | 它是不是看到别人忽略的限制？比如数据不能出内网 |
| 真入口 | 它有没有一个用户马上能理解的小入口？           |
| 真趋势 | 它是不是站在一个更大的变化上？                 |

## 输出模板

```
## 它做对了什么

1. 它抓住了一个真问题：
   【一句话说明】

2. 它选择了一个正确约束：
   【例如私有化部署 / 低成本 / 本地化 / 垂直场景】

3. 它的第一层入口是清楚的：
   【用户第一步能用它干什么】

4. 它背后可能对应一个趋势：
   【可迁移的大方向】
```

## 例子：Coomia 类产品

不是只说“AI 问数”。

更应该看到：

> 它抓住的是“企业想用 AI 分析数据，但数据不想出内网”这个真实约束。

这个趋势不是孤例。GoodData 也在 2025 年宣布把 AI Assistant 和 Smart Search 带到 self-hosted deployment，强调企业可以在自己的基础设施或私有云中运行 AI 分析能力，避免第三方接触数据。

# Step 4：找外部误解 / 叙事错位

## 目标

找出创始人自己不容易看到的“后脑勺”。

## 五类常见错位

| 错位类型   | 典型表现                                       |
| ---------- | ---------------------------------------------- |
| 产品名错位 | 官网叫 A，文档叫 B，用户不知道是不是同一个产品 |
| 阶段错位   | 产品还在工具阶段，首页已经在讲平台             |
| 客户错位   | 文案吸引大企业，但价格和交付更像小团队工具     |
| 证据错位   | 讲的是平台能力，证据只有 demo / GitHub / 文档  |
| 交付错位   | 轻工具价格，承诺重平台价值                     |

## 必答问题

1. 官网、文档、定价、GitHub 是否在讲同一个产品？
2. 用户第一眼能不能知道“现在能用什么”？
3. 长期愿景是否被写成了当前事实？
4. 现在最可信的能力是哪一层？
5. 哪个词最可能让用户产生过高预期？

## 输出模板

```
## 外部误解风险

从外部视角看，这个产品可能被理解成两个东西：

第一层：
【现在最可信、最容易试用的入口】

第二层：
【团队更长期、更大的愿景】

风险不在于两者冲突。
风险在于两者关系没有讲清楚。

用户可能会问：
“我现在买到的，到底是【A】，还是【B】？”
```

# Step 5：判断商业模式与交付风险

## 目标

看“卖法”和“做法”是否匹配。

## 核心问题

1. 如果是轻工具，是否能低交付？
2. 如果是平台，是否有足够客单价支撑重交付？
3. 私有化部署会不会让每个客户都变成定制项目？
4. 年费能否覆盖销售、部署、客户成功和支持成本？
5. 有没有办法把第二个客户的交付成本降下来？

## 风险表

| 风险           | 判断句                                 |
| -------------- | -------------------------------------- |
| 低价私有化陷阱 | 年费不高，但每个客户都要部署和支持     |
| 平台叙事陷阱   | 客户按平台预期购买，团队按工具能力交付 |
| 重交付泥潭     | 每单都要定制，没有模板复用             |
| 销售错配       | 文案吸引大客户，但团队承接不了         |
| 自助失败       | 说是工具，但用户无法自助部署和使用     |

## 输出模板

```
## 商业模式风险

当前最需要警惕的是：
【某个具体风险】

原因：
【为什么这个风险会出现】

最小修正：
【降低交付 / 改定价 / 改定位 / 增加模板 / 明确边界】
```

# Step 6：抽象可迁移结构

## 目标

不能只停在“这个产品怎么样”。

要问：

> 它背后有没有一套可迁移到其他行业的结构？

## 提取方法

把产品拆成 5 个要素：

```
用户：
约束：
旧方案：
新能力：
可迁移结构：
```

## 输出模板

```
## 可迁移结构

这个产品表面上做的是：
【具体产品】

但更底层的结构是：
【抽象结构】

这个结构可以迁移到：
【行业 / 场景 / 人群】
```

## 例子

```
表面产品：
自托管 AI 数据分析工具。

底层结构：
把原来需要数据团队 / 重型 BI / 企业平台才能完成的分析能力，
压缩成一个中小团队也能私有化部署的小系统。

可迁移方向：
高价企业软件能力的轻量化、私有化、平民化。
```

## 注意

不要直接说：

> 做一个国产 Palantir。

Palantir 的 Ontology 官方定义很重，它不只是语义层，而是组织的 operational layer，连接 datasets、models 和真实世界对象，还包含 actions、functions、dynamic security 等。

所以更稳的说法是：

> 学它的结构，不学它的口号。
>  从一个窄入口开始，而不是一上来做平台。

# Step 6.5：时间变量推敲

## 目标

判断这个产品或可迁移机会是否会被未来 **3–5 年**的技术、成本、监管、用户习惯变化 **打开、压缩、改写或杀死**。

## 必答问题

1. 这个机会是否依赖当前技术成本较高或能力不足？
2. 若 AI 推理、多模态、端侧部署、检测设备、机器人或生物技术进一步成熟，会发生什么？
3. 未来 3–5 年是否存在会 **杀死** 这个机会的变量？
4. 若变量发生，这个机会应 **如何改写**？
5. **当前还值得做的最小切口**是什么？

## 输出模板

### 时间变量推敲

当前机会：
【一句话】

未来变量：
1. 【变量A】
2. 【变量B】

影响判断：
- 变量A 会【打开 / 压缩 / 改写 / 杀死】机会；
- 变量B 会【打开 / 压缩 / 改写 / 杀死】机会。

修正后的机会表述：
【不要写成永恒机会，而是写成当前阶段仍然成立的具体切口】

## 铁律

时间变量只用于 **校准**，不用于制造焦虑。

不能因为未来可能变化，就否定当前真实痛点。

也不能因为当前痛点存在，就忽略未来可能杀死机会的变量。

# Step 7：做小型深度研究

## 目标

把“我觉得有几个方向”变成“我按筛子筛过几个方向”。

## 触发条件

只要文章里出现：

> 这个结构可以迁移到很多行业。

就必须做一轮小型深度研究。

否则会显得虚。

## 研究问题模板

```
## 研究问题

从【某产品】拆解出发，验证：

“【可迁移结构】”在哪些垂直行业更容易成立？
```

## 筛选条件

建议默认 6 条：

1. 数据敏感；
2. 现有软件太贵 / 太重 / 太复杂；
3. 工作高频；
4. AI 输出可被人复核；
5. 可以从小入口开始；
6. 3-6 个月内可能出现付费信号。

## 候选方向表

```
| 方向 | 数据敏感 | 高频程度 | 现有软件痛点 | AI可复核 | 小入口清晰 | 付费可能 | 总评 |
|---|---|---|---|---|---|---|---|
| 方向A | 高 | 高 | 高 | 高 | 高 | 中高 | 优先 |
| 方向B | 中 | 高 | 中 | 高 | 高 | 中 | 可看 |
| 方向C | 高 | 中 | 高 | 中 | 中 | 中 | 谨慎 |
```

## 输出要求

深度研究不要只列方向。
 必须选出一个“最适合正文展开”的方向。

# Step 8：收束到一个具体方向

## 目标

公众号正文里不要堆 7 个方向。

正文只展开一个方向。
 其他方向放附件。

## 选择标准

优先选：

1. 和原产品启发最近；
2. 用户场景最接地气；
3. 能用一句话讲清楚；
4. 数据和证据较容易补；
5. 能 3-6 个月做 MVP 验证。

## 输出模板

```
## 最值得先看的方向

方向：
【具体方向】

一句话：
帮【具体用户】，在【具体场景】，用【AI能力】，解决【具体问题】。

为什么是它：
1. 【原因1】
2. 【原因2】
3. 【原因3】

最小 MVP：
【第一版只做什么】

Kill 条件：
【什么情况说明不值得继续】
```

## 例子

```
方向：
小制造业经营数据 Copilot。

一句话：
帮小制造业老板在自己的数据环境里，直接问订单、库存、回款和利润问题。

最小 MVP：
接入订单、库存、应收应付三类数据，预设 20 个老板常问问题。

Kill 条件：
数据接不进来；老板不愿给数据；问题不是高频；答案无法被信任。
```

# Step 9：生成四类输出

一次拆解最好生成 4 个东西。

## 输出 A：私下建议稿

给产品团队看。

结构：

```
1. 我先看到你们做对了什么
2. 外部用户可能在哪里误解你们
3. 这个误解为什么重要
4. 三个最小修正动作
5. 我愿意继续验证的问题
```

语气：

- 不说“你们错了”；
- 说“外部用户可能会这样理解”；
- 不否定愿景；
- 只建议先把第一层讲清楚。

------

## 输出 B：公众号文章

给公众看。

结构：

```
1. 我拆了一个小团队产品
2. 它抓住了一个真问题
3. 它最值得学的地方是什么
4. 它可能存在什么外部误解
5. 从它身上推演出一个可迁移结构
6. 深度研究后，我选了一个具体方向
7. 如果你也有产品，可以发给我
```

公众号正文原则：

- 不堆太多理论；
- 不列太多方向；
- 一个案例 + 一个判断 + 一个方向；
- 用“这点我真没想到”式洞察打动读者。

------

## 输出 C：深度研究附件

给有兴趣的人看。

结构：

```
# 附件：可迁移方向初筛报告

## 1. 研究问题
## 2. 筛选条件
## 3. 候选方向
## 4. 对比评分
## 5. 为什么优先选这个方向
## 6. MVP 假设
## 7. Kill 条件
## 8. 待验证问题
```

------

## 输出 D：产品招募语

放文章结尾。

示例：

```
如果你也在做一个产品，可以发给我。

我想用 SK 做三件事：

第一，看你已经做对了什么。
第二，看外部用户第一眼会不会误解你。
第三，从你的产品里推演更大的机会。

我不一定比你懂你的产品。

但我可以站在外部用户的位置，看一眼你的“后脑勺”。

如果我看错了，你可以纠正我。
如果我看对了一点，也许它能帮你少绕一段路。
```

------

# Step 10：决定公开、脱敏或点名

## 决策表

| 情况                   | 发布方式             |
| ---------------------- | -------------------- |
| 对方未同意             | 完全脱敏             |
| 对方同意但不想暴露问题 | 写共创版，少写负面词 |
| 对方愿意公开讨论       | 点名版               |
| 涉及敏感数据           | 不公开               |
| 只是用来推演方向       | 脱敏即可             |

## 推荐原则

熟人产品默认先私发。
 公开文章默认脱敏。
 点名前必须取得同意。

## 点名版必须加一句

```
这不是对该产品的定论。我没有深度使用产品，只是基于公开信息做一次外部视角校准。文章发布前，也把事实部分发给团队看过，避免基本信息误读。
```

# 最终交付模板

下面是每次拆解可以直接复制的模板。

```
# 【产品名】外部体检与机会推演

## 0. 本次边界

- 是否公开：
- 是否点名：
- 是否取得同意：
- 资料来源：
- 是否实测：
- 输出目的：

## 1. 产品一句话

它帮【谁】，在【什么场景】，解决【什么具体问题】。

## 2. 公开资料包

- 官网：
- 文档：
- GitHub：
- 创始人内容：
- 定价：
- 客户案例：
- 证据缺口：

## 3. 它做对了什么

1. 真问题：
2. 真约束：
```

# 本 SOP 的核心铁律

1. **先看优点，再看问题。**
2. **先找第一层价值，再谈长期愿景。**
3. **先拆真实产品，再推演可迁移方向。**
4. **公众号只写一个具体方向，深度研究放附件。**
5. **熟人产品默认脱敏，点名前先征求同意。**
6. **不把未来路线图当成当前事实。**
7. **不做情绪批评，只做外部误解校准。**
8. **每次拆解都要产出一个“这点我真没想到”。**
