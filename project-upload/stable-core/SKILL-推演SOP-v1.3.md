# SKILL：从模糊想法到完整商业模型的推演SOP

> 版本：v1.3（v1.0八步流程，v1.1灵感激发模式，v1.2三个纪律+北极星，v1.3底层能力：透过现象看本质七步法）
> 适用场景：有人带着一个模糊的产品想法来，需要判断值不值得押注
> 预计时长：单次完整推演 2-4 小时
> 输出物：流水账式推演记录（.md），不给固定结论，保留所有细节
> 后续衔接：推演结论为"做"时，进入「SKILL-产品设计方法论」

---

## 使用前提

1. 推演者必须有至少一个**真实接触过的相关案例/人物**（不能纯靠桌面研究）
2. 必须搜知识库再动手（推演公式、财富公式、失败模式库、维度跨越理论）
3. 流水账全程记录，不删任何"走弯路"的内容——弯路本身就是迭代素材

## 底层能力：透过现象看本质七步法（v1.3新增）

这七步不是推演SOP的某一步，而是**每一步都在用的底层能力**。没有它，九步推演只是流程；有了它，每一步都能产出真正的洞察。

灵感激发模式是系统帮推演者做的事。七步法是推演者自己必须具备的认知操作系统。

### 为什么需要这个

推演过程中最大的陷阱是：数据/框架/AI给了一个"看起来对"的答案，推演者就信了。

老年AI管家推演中，这个陷阱至少出现了五次：

| 时刻 | "正确的答案" | 推演者的真实观察 |
|------|------------|----------------|
| 竞品分析后 | 老人需要陪伴 | 那对73岁夫妻不孤独，他们要的是被看见 |
| 产品设计v0.1 | 用记忆/网络/成就锁住用户 | 不是锁住，是服务好到没理由走 |
| 视频策略讨论 | 应该做多平台视频矩阵 | 我在做流量不是在发信号 |
| 写001时 | 先自我介绍再拆解 | 读者不关心你是谁，关心你有没有一句他没想到的话 |
| GPT深度研究后 | 防诈骗场景最强，应该先做 | 被骗的人不知道自己被骗，防诈骗是低频场景 |

每一次都是同一个动作：**用真实的人推翻"看起来对"的结论。** 这个能力才是推演的核心，九步流程只是它的脚手架。

### 七步法

**第零步：从一个真实的人开始**

不是从概念开始，不是从数据开始，不是从理论开始。是从一个你认识的、你观察过的、你跟他聊过天的真实的人开始。

没有那对73岁夫妻，就没有"老人要的不是陪伴"这个洞察。没有你自己作为读者的真实感受，就没有"先拆解不要自我介绍"这个判断。

**检验：如果你的推演里找不到一个有名有姓（或者你能描述出他长什么样、住哪里、平时干什么）的真实的人，推演大概率会停在"正确但无用"的结论上。**

参考理论：透过现象看本质（冰山模型）——看到的行为只是冰山一角，海面下是需求、动机、信念。

**第一步：发现跟"大家都以为的"不一样的东西**

观察那个真实的人，找到一件跟"行业共识""常识""数据显示"不一样的事。

"所有人都说老人需要陪伴。但我观察到的那对夫妻根本不缺陪伴。"

这一步的本质是反证/证伪——不是去证明通用结论是对的，而是去找它不成立的地方。只要找到一个真实的反例，就值得深挖。

参考理论：不要正面回答别人的问题——不回答"问题本身"，回答"问题背后的那个人"。行业说"老人需要陪伴"，这是"问题本身"；真实的老人在做什么，这才是"问题背后的那个人"。

**第二步：问"他真正要的到底是什么"**

找到了"跟常识不一样的东西"之后，追问根本需求。

"他们不需要陪伴。那他们到底要什么？"——观察他们最开心的时刻，发现是"在别人眼中看到自己存在的价值"。

不是停在"他们不需要陪伴"（否定别人的答案），而是继续走到"他们需要被看见"（找到自己的答案）。

参考理论：超越用户的预期——"用户知道自己不想要什么（痛点），但不一定知道自己真正想要什么（渴望）。"

**第三步：检查自己是不是在用设计者视角**

时刻反省：我是不是还在用产品设计者的视角，而不是用户的视角？

"防诈骗数据最强，应该先做"——这是设计者视角（哪个场景数据好就先做哪个）。

"一个老人一天到晚去防诈骗？"——这是用户视角（这件事在他的生活里占多大比重？）。

参考理论：不要正面回答别人的问题——回应"那个人"，不是回应"那个问题"。GPT深度研究告诉你"防诈骗排第一"，这是"问题"；老人每天的真实生活里有多少时间在想防骗，这才是"那个人"。

**第四步：重新定义问题**

如果前三步发现需求变了，问题也得重新定义。

原来的问题："怎么做一个陪伴老人的AI？"
重新定义后的问题："怎么让老人用上AI，从而生活真的变好？"

竞争对手也跟着变了。原来的竞争对手是Hyodol、ElliQ。重新定义后的竞争对手变成了"所有让老人觉得自己没被时代抛下的东西"。

参考理论：维度跨越（DOC-D021）、换题思维（DOC-S074）。

**第五步：频率测试**

对任何结论，问一个问题："这件事在一个真实的人的真实生活里，每天/每周发生几次？"

数据说"25.8万起电诈案件"——很大。但翻译成频率："一个具体的老人，一个月可能遇到一次可疑短信"——很低。

数据说"老人社交孤立很严重"——很大。但翻译成频率："那对73岁夫妻每周跟俱乐部朋友见三次面"——对这批人来说不成立。

**频率测试是对抗"数据幻觉"的武器。** AI和研究报告最容易犯的错误是用"总量"吓你。推演者要把总量翻译成"一个具体的人每天的真实体验"。

高频 = 可以做入口/习惯。低频 = 只能做附加值/偶发场景。

**第六步：想到不同的角度**

这一步不是"为了不同而不同"。是因为前五步已经把你带到了一个别人到不了的位置——从那个位置看过去，自然会看到别人看不到的东西。

"入口是虚荣的没关系，只要用了之后生活真的变好了。"——这个判断不是"想"出来的，是从73岁夫妻的观察（第零步）→ 发现他们不需要陪伴（第一步）→ 找到"被看见"的需求（第二步）→ 检查用户视角（第三步）→ 重新定义问题（第四步）→ 测试频率（第五步）一步步"走"到的。

**如果跳过前五步直接"想一个不同的角度"，大概率想出来的是聪明但没用的观点。**

### 七步法在推演SOP每一步中的应用

| 推演步骤 | 七步法的作用 |
|---------|------------|
| 第一步（接住模糊想法） | 第零步：确保有真实的人作为锚点 |
| 第二步（对标扫描） | 第一步：对标找到了，但跟你观察到的真实情况一样吗？ |
| 第三步（真实案例纠偏） | 第一步+第二步：用真实案例推翻对标分析的结论 |
| 第四步（重新定义问题） | 第三步+第四步：检查视角+重新定义 |
| 第五步（跑历史数据） | 第五步：数据给出的结论，频率测试能不能过？ |
| 第六步（设计增长机制） | 第三步：这个增长机制是设计者觉得对的，还是用户真的会做？ |
| 第七步（商业模型设计） | 第三步+纪律三：是为用户好还是为自己好？ |
| 第八步（沉淀） | 记录哪些时刻用了七步法，哪些结论被推翻了 |
| 第九步（北极星） | 第二步+第四步：北极星说的是用户的根本需求，不是产品的功能 |

### 与灵感激发模式的关系

灵感激发模式的7个角度是系统帮推演者"制造认知触发事件"。七步法是推演者自己"判断这个触发是不是真的有价值"。

系统说"历史上最像的是XXX"（角度1·历史类比）→ 推演者用七步法判断："但我观察到的真实情况不是这样的"（第一步：发现不一样的东西）→ AHA出现。

**没有七步法，推演者会接受系统给的每一个"看起来对"的答案。有了七步法，推演者有能力说"不对"。**

### 实战验证

老年AI管家推演中，七步法的五次关键应用：

1. **"老人要的不是陪伴，是被看见"**
   - 第零步：73岁夫妻
   - 第一步：他们不孤独（跟"老人需要陪伴"不一样）
   - 第二步：他们最开心的时刻是被邻居认可
   - 第六步：入口可以是虚荣的，只要结果是真的

2. **"不是锁定，是服务"**
   - 第三步：检查视角——"我在设计怎么困住用户，这是设计者视角"
   - 第四步：重新定义——从"怎么锁住"到"怎么服务好到没理由走"

3. **"做流量还是发信号"**
   - 第三步：检查视角——"我在讨论多平台视频矩阵，这是做流量不是发信号"
   - 第五步：频率测试——"我的目标读者每天看几条视频号？"→回到北极星

4. **"先拆解不要自我介绍"**
   - 第零步：自己作为读者的真实体验
   - 第三步：检查视角——读者想知道什么 vs 作者想说什么
   - 第四步：重新定义——001的任务从"让人认识我"变成"让人信任我的判断"

5. **"防诈骗不是入口"**
   - 第零步：我认识的老人没有整天想着防骗
   - 第一步：被骗的人不知道自己在被骗（跟"防诈骗需求最大"不一样）
   - 第五步：频率测试——一个老人一个月遇到一次可疑短信 = 低频

## 核心机制：灵感激发模式

### 人机分工原则

八步推演中，有两类工作：

**系统做的（需要纪律，不需要灵感）：**
搜索对标、跑历史数据、检查铁律清单、标注证据等级、沉淀流水账、提取资产存入知识库。人会偷懒跳步，系统不会。

**人做的（需要灵感，不需要纪律）：**
给出真实案例、纠偏假设、跨领域类比、重新定义问题、判断"值不值得押注"。系统想不到"老年人=固化了的儿童"，人能。**推演者使用七步法来判断系统给出的答案是不是真的成立。**

**四个人类灵感节点：**

| 节点位置 | 系统已完成的事 | 等待人输入的东西 |
|---------|-------------|--------------|
| 第二步之后 | 自动搜索出5-8个对标 | 补充遗漏的对标/跨领域类比 |
| 第三步 | 呈现相关历史案例 | 给出真实案例+纠偏假设 |
| 第四步 | 提供维度跨越检查清单 | 决定新的问题定义 |
| 第七步之后 | 列出商业模型和防御设计 | 判断"值不值得押注" |

### 灵感激发模式（v1.1新增）

**在每个人类灵感节点，如果推演者没有立即给出输入，系统不是"暂停等待"，而是进入灵感激发模式。**

系统从以下7个标准角度逐一展开提示，制造认知触发事件（CTCE），直到推演者说"这个有意思"（AHA出现）或"先往下走"（标记此步未获灵感突破）：

**角度1·历史类比：**
"历史上跟这个最像的产品/模式是什么？它做对了什么？死在了哪？"

**角度2·反转假设：**
"如果你假设的用户根本不是真正的用户呢？如果付费方不是你以为的那个人呢？"

**角度3·真实案例追问：**
"你身边有没有人正在经历这个问题？他是怎么解决的？他的'笨办法'是什么？"

**角度4·维度跨越：**
"如果给这个问题加一个维度（时间/空间/社交/情感），问题变成什么？"

**角度5·极端推演：**
"如果这个产品10倍成功了，世界会变成什么样？如果它彻底失败了，最可能是因为什么？"

**角度6·对手视角：**
"如果你是腾讯/字节/美团，你怎么做这件事？你做不到什么？"

**角度7·用户的话：**
"如果用户只能用一句话描述为什么用你的产品，他会说什么？如果这句话里没有你的产品名，说明什么？"

### 运行规则

1. 每展开一个角度，等推演者反应
2. 推演者可以在任何角度打断："这个有意思"→ 深入展开 → AHA
3. 推演者也可以被某个角度激活出完全不同的想法："你刚才说的让我想到了——" → 更好的AHA
4. 7个角度全部展开后推演者仍无灵感 → 标记"此步未获灵感突破"，继续下一步
5. **记录哪个角度触发了AHA** → 积累数据后优化角度排序（高频触发的排前面）
6. **推演者用七步法检验AHA的真实性** → 不是所有AHA都是洞察，有些只是"听起来聪明"（v1.3新增）

### 认知原理

系统展开的7个角度不是在给答案，是在制造认知触发事件。AHA时刻的本质是"推演者脑子里本来就有的两个东西突然连上了"。系统的作用是提供足够多的"连接催化剂"。

**七步法是推演者判断"这个连接是真实的还是虚假的"的能力。** 不是每个AHA都值得追。值得追的AHA通常可以用七步法回溯到一个真实的人（第零步）和一个跟常识不一样的观察（第一步）。（v1.3新增）

灵感激发不是随机的，是可以被优化的。优化方式不是"给更好的答案"，而是"问更好的问题"。

### 首次实战验证

老年AI管家推演中，AHA的触发模式：
- "老年人=固化了的儿童" → 由角度3（真实案例追问）触发
- "不是陪伴是社交，小天才的逻辑" → 由角度1（历史类比）触发
- "不是给自己做，是送给别人" → 由角度7（用户的话）的变体触发
- "安装=加微信好友，机构的人=地推" → 由角度2（反转假设）触发

### 灵感激发的迭代方向

- v1.1（当前）：7个标准角度，手动逐一展开
- v2.0：根据推演主题自动选择最相关的3-4个角度优先展开（不是全部跑完）
- v3.0：积累20次以上AHA记录后，训练"角度推荐模型"——根据想法类型预测哪个角度最可能触发AHA
- v4.0：角度库开放扩展——每次推演中如果出现了不在7个标准角度内的新触发方式，自动提取为第8、第9个角度

## 三个纪律（v1.2新增）

### 纪律一：多AI讨论融合

同一个想法，可以跟不同的AI（Claude/KIMI/GPT/DeepSeek）讨论。不同AI的知识结构和追问风格不同，会把想法推向不同方向。

**操作方式：**
- 跟AI-A做完整推演（商业模型、竞品分析）
- 跟AI-B做底层追问（人性洞察、哲学反思）
- 把两份讨论的结论做对照表，找出共识和分歧
- 分歧处就是需要创始人自己做判断的地方

**本次实战验证：**
- Claude讨论侧重"怎么做"——商业模型、竞品分析、裂变机制
- KIMI讨论侧重"为什么做"——人性底层逻辑、"意大利鞋"等式、普朗克陷阱
- 两者不矛盾，是不同层次。融合后产品设计比任何单一讨论都完整

**注意：不是让AI之间辩论，是让AI各自展开，创始人做裁判。** AI不知道另一个AI说了什么，创始人是唯一知道全貌的人。

### 纪律二：灵感纪律——灵感≠结论

推演过程中会冒出很多灵感。灵感很兴奋，容易让人立刻把它当结论。

**规则：灵感出现时，做三件事：**
1. 记录下来（不要丢）
2. 标记为"灵感·待验证"（不要当结论）
3. 列出验证方式（怎么证明这个灵感是对的或错的）

**本次实战验证：**
讨论中出现"数字人视频通话"的灵感。如果当场就写进产品设计文档，可能浪费大量时间做一个老人根本不想要的东西。实际处理：标记为灵感→列出技术评估清单→设计小步快跑验证路径→先不改产品设计文档。

**灵感的价值不在于它"是对的"，在于它"可能是对的"。** 对待灵感的方式是：认真对待，但不轻信。

### 纪律三：价值观校准——是为用户好，还是为自己好？

推演到商业模型和留存设计时，很容易滑入"怎么锁住用户"的思维。

**在第七步（商业模型与防御设计）完成后，必须做一次价值观校准：**

问自己："这个设计，是让用户的生活更好？还是只是让用户走不了？"

| 锁定思维（自私） | 服务思维（亚马逊式） |
|----------------|-------------------|
| 怎么提高迁移成本 | 怎么让服务好到没理由走 |
| 怎么设计成就系统留住人 | 怎么让用户真的成长 |
| 怎么用社交网络绑住人 | 怎么帮用户维护真实关系 |
| 怎么让免费版不够用 | 怎么让免费版就很好用 |

**结果可能一样（用户不走），但出发点完全不同。** 用户能感受到区别。

**本次实战验证：**
产品设计v0.1的三层锁定（记忆锁定/网络锁定/成就锁定）在v0.2中被翻转为三层服务深度。出发点从"困住用户"变成"服务好用户"。

## 第一步：接住模糊想法，不急着定义

### 做什么

收到想法后，**原样记录**，不替对方补充、不替对方纠偏、不替对方美化。

记录三样东西：
- 他说了什么（原话）
- 他没说什么（哪些关键信息缺失）
- 他的语气暗示什么（兴奋？试探？焦虑？）

### 不做什么

- 不立刻给判断（"这个方向不错"或"这个很难"）
- 不立刻进入分析模式
- 不替他定义用户、定义市场、定义商业模型

### 输出

一段原始记录。3-5句话。保持模糊。

### 本次实战示例

> 缺失信息：谁付费？什么类型的机构？老年人的具体画像？
> 语气：探索性的，"只是一个简单的设想"

## 第九步：确立北极星（v1.2新增）

### 做什么

八步推演完成后，如果结论是"做"，在进入产品设计之前，先用一句话确立北极星。

**北极星 = 这个产品存在的唯一理由。** 后续所有产品决策都用这句话过滤。

### 怎么写

北极星必须满足三个条件：
1. **一句话**——不超过20个字
2. **说清楚为谁、做什么、结果是什么**
3. **不包含任何具体功能或形式**（功能和形式会变，北极星不变）

### 检验方法

用北极星过滤所有产品决策。如果某个决策跟北极星无关或矛盾，就不做。

**示例（老年AI管家）：**

北极星：让老年人用上AI，从而生活变得更好。

| 决策 | 过滤 |
|------|------|
| 做数字人视频通话？ | 如果它让老人更容易用上AI，做；如果只是炫酷，不做 |
| 做成就系统？ | 如果它让老人用得更深，做；如果只是留存手段，不做 |
| 定价99元/月？ | 如果老人觉得生活真的变好了，99元他不会犹豫 |

### 输出

- 北极星文档（独立文件，不放在产品设计文档里，因为产品设计会迭代但北极星不动）
- 文件命名：`north-star-[项目名].md`

### 本次实战示例

> 北极星：让老年人用上AI，从而生活变得更好。
> 文件：north-star-elderly-ai.md

## 附录E：推演SOP与产品设计方法论的关系（v1.2新增）

```
推演SOP（本文件）              产品设计方法论（SKILL-产品设计方法论.md）
       |                                    |
  "做不做"的判断               "怎么做"的执行
       |                                    |
  第一步→第八步                第一步→第N步
       |                                    |
  结论："做" ──→ 第九步确立北极星 ──→ 进入产品设计方法论
  结论："不做" ──→ 存档，不进入产品设计
```

推演SOP回答"值不值得做"。产品设计方法论回答"决定做之后，怎么从想法到可执行产品"。

北极星是两者的桥梁——推演结束时确立，产品设计时作为所有决策的过滤器。

**七步法贯穿两者——无论在判断"做不做"还是设计"怎么做"，每一个关键结论都需要经过七步法的检验。**（v1.3新增）
